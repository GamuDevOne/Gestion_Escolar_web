Tarea: Página de cambio de contraseña obligatorio
Debes crear una página cambiar_password.html en la carpeta frontend/ que se muestre automáticamente cuando un usuario inicia sesión por primera vez
con su contraseña inicial. 
Esta página debe ser obligatoria — el usuario no puede saltársela ni acceder a su vista normal sin completarla.

Lo que debe tener la página:
Un formulario con tres campos:

Contraseña actual
Contraseña nueva
Confirmar contraseña nueva

Y un botón de guardar. El diseño debe seguir el estilo visual de login.css / index.html ya que es una página independiente del dashboard.

Archivos que tienes disponibles para trabajar:

frontend/css/login.css — usa este CSS como base para el diseño de la página
frontend/js/api.js — ya tiene API_URL y apiFetch listos para usar, inclúyelo antes de tu JS
frontend/js/login.js — referencia de cómo se hace un fetch al backend con manejo de errores


Cómo debe funcionar el JS de la página:

1. Al cargar la página, leer el localStorage para obtener el token y el rol del usuario
   - Si no hay sesión activa, redirigir a index.html
   - Si password_cambiada es true, redirigir directo a su vista (no necesita cambiar)

2. Al enviar el formulario:
   - Validar que los campos no estén vacíos
   - Validar que "contraseña nueva" y "confirmar contraseña" coincidan
   - Validar que la nueva contraseña tenga al menos 6 caracteres
   - Hacer un POST a: /auth/cambiar_password.php
     con el body: { password_actual, password_nueva }
     y el header Authorization: Bearer <token> (apiFetch lo hace automáticamente)

3. Si el servidor responde success: true
   - Actualizar en localStorage: password_cambiada = true
   - Mostrar mensaje de éxito (puede usar SweetAlert2 que ya está disponible)
   - Redirigir según el rol guardado en localStorage:
     admin     → admin.html
     profesor  → profesor.html
     estudiante → estudiante.html

4. Si el servidor responde error
   - Mostrar el mensaje de error que devuelve el servidor en json.error

Estructura de scripts al final del HTML:
<script src="https://cdn.jsdelivr.net/npm/sweetalert2@11"></script>
<script src="js/api.js"></script>
<script src="js/cambiar_password.js"></script>

Lo que devuelve el backend en cada caso:

Éxito:
json{ "success": true, "data": { "message": "Contraseña actualizada correctamente" } }

Error contraseña actual incorrecta:
json{ "success": false, "error": "La contraseña actual es incorrecta" }

Error contraseña igual a la anterior:
json{ "success": false, "error": "La nueva contraseña no puede ser igual a la actual" }

Lo que NO debes tocar — ya está hecho por el backend:

La lógica de validación de contraseñas está en el servidor
El token se maneja automáticamente con apiFetch
La columna password_cambiada en la base de datos ya existe
La redirección automática al login si el token expira ya la maneja api.js

-Flujos adicionales de contraseña!
estas páginas/flujos se agregan una vez esté lista cambiar_password.html.

1. Página de configuración de preguntas de seguridad
Crear frontend/configurar_preguntas.html
Esta página aparece automáticamente después de que el usuario cambia su contraseña por primera vez. Es obligatoria igual que el cambio de contraseña, no se puede saltar.

Lo que debe tener:
Un formulario con tres grupos, cada uno con un <select> para elegir la pregunta y un <input> para escribir la respuesta. Las preguntas disponibles las devuelve el backend. Y un botón de guardar.

Cómo debe funcionar el JS:

1.Al cargar, verificar sesión en localStorage igual que cambiar_password.html. Si no hay sesión, redirigir a index.html.
2.Hacer un GET a /auth/preguntas_seguridad.php para obtener el listado de preguntas y poblar los tres <select>.
3.Al enviar el formulario, validar que ningún campo esté vacío y que no se repita la misma pregunta en los tres grupos.
4.Hacer un POST a /auth/configurar_preguntas.php con el body:
{ 
  "preguntas": [
    { "pregunta_id": 1, "respuesta": "..." },
    { "pregunta_id": 3, "respuesta": "..." },
    { "pregunta_id": 5, "respuesta": "..." }
  ]
}

5.Si el servidor responde success: true, redirigir según el rol igual que en cambiar_password.html.

El flujo completo del primer login queda así:
index.html → cambiar_password.html → configurar_preguntas.html → vista del rol

Estructura de scripts al final del HTML:
<script src="https://cdn.jsdelivr.net/npm/sweetalert2@11"></script>
<script src="js/api.js"></script>
<script src="js/configurar_preguntas.js"></script>

2. Flujo de recuperación de contraseña (olvidé mi contraseña)
No es una página separada, se maneja desde index.html.
Agregar debajo del formulario de login un enlace: ¿Olvidaste tu contraseña? que muestre un flujo de tres pasos usando SweetAlert2 (sin salir de la página):

Paso 1: Input para ingresar el email o identificación del usuario. Al confirmar, hace GET a /auth/preguntas_usuario.php?identificacion=... y obtiene las 3 preguntas de ese usuario.
Paso 2: Muestra las 3 preguntas con inputs para las respuestas.
Paso 3: Input para la nueva contraseña y confirmación.

Al enviar el paso 3, hace POST a /auth/recuperar_password.php con el body:
{
  "identificacion": "...",
  "respuestas": ["...", "...", "..."],
  "password_nueva": "..."
}
Si el servidor responde success: true, mostrar mensaje de éxito y quedar en el login para que el usuario entre con su nueva contraseña.

3. Cambio de contraseña desde dentro de la app
Tanto admin, profesor como estudiante deben tener en su vista respectiva una opción para cambiar contraseña voluntariamente.
Puede ser un botón en el sidebar o en el header que abra un modal con el mismo formulario de cambiar_password.html (contraseña actual, nueva, confirmar). Usa el mismo endpoint: POST /auth/cambiar_password.php. No redirige a ningún lado al completarse, solo cierra el modal y muestra éxito con SweetAlert2.

4. Reset de contraseña por el admin
En la vista de admin, al editar un estudiante o profesor, ya existe el campo de contraseña inicial. Solo hay que agregar un botón visible tipo Resetear acceso que al presionarse haga la edición con la nueva contraseña y adicionalmente le avise al admin que el usuario deberá volver a hacer el proceso completo de configuración la próxima vez que inicie sesión.
No requiere página nueva, todo dentro del modal de edición existente.

Archivos nuevos que necesita crear tu compañera:

frontend/configurar_preguntas.html
frontend/js/configurar_preguntas.js
Modificaciones a frontend/index.html y frontend/js/login.js para el flujo de recuperación
Modificaciones menores a admin.js para el reset y a cada vista para el cambio voluntario de contraseña

Estilos: Todo debe seguir login.css para las páginas independientes y el CSS del rol correspondiente para los modales dentro de la app.
